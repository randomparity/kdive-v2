# M1.4 — System catalog, availability & scheduling (Integration Contract)

## Purpose

M1.4 realizes the latent domain-model concepts the top-level design named but M0–M1.3
never built: named system **shapes** over the provisioning profile, **full custom**
configuration including PCIe devices, a **fleet availability** view, a **reservation /
backlog scheduler** so a scarce host is never idle with work waiting, and explicit
**system reuse**. It is the last entry in the M1.x local-libvirt feature-deepening band
before the provider-expansion milestones (M2+); it stays on the single `local-libvirt`
provider. It adds tools and one new durable subsystem (the pending queue), and it **does
change admission-core logic** — the occupancy-count predicate, PCIe-aware host selection,
the allocation schema and state machine — but it **does not touch the provider interface**:
adding M2's remote provider still touches zero provider-seam lines.

- **Decisions:** [ADR-0067](../adr/0067-system-shapes-catalog.md) (shapes catalog +
  selector unification), [ADR-0068](../adr/0068-custom-config-pcie-modeling.md) (custom
  config + PCIe capability modeling), [ADR-0069](../adr/0069-reservation-pending-queue-scheduler.md)
  (reservation / FIFO pending-queue scheduler),
  [ADR-0070](../adr/0070-fleet-availability-system-reuse.md) (fleet availability + system
  reuse). All build on [ADR-0007](../adr/0007-metering-budgets-admission.md) (the
  selector / admission gate the size and reservation attach to),
  [ADR-0023](../adr/0023-discovery-allocation-admission.md) (the per-host capacity check),
  [ADR-0036](../adr/0036-reservation-lease-semantics.md) (the lease window a granted
  reservation acquires), [ADR-0021](../adr/0021-reconciler-loop-drift-repair.md) (the
  reconciler the promotion sweep joins), and [ADR-0062](../adr/0062-platform-operations.md)
  (the `cordoned` schedulability flag and the runtime-knob operator-tool pattern the
  shapes tools mirror).
- **Parent:** [`top-level-design.md`](top-level-design.md) §Roadmap (M1.4).

## What M1.4 adds

- **A system-shape catalog** — a `system_shapes` table seeded by migration with named
  presets (`small` … `max`), each a sizing tuple `{vcpus, memory_mb, disk_gb}` plus an
  optional PCIe-device requirement. `platform_operator` tools tune the catalog at runtime
  (`shapes.set` / `shapes.delete`), mirroring M1.3's `ops.set_host_capacity`; a `viewer`
  `shapes.list` reads it. The allocation selector accepts a **shape name XOR a full-custom
  sizing**, and the resolved size flows into both admission (cost/capacity) and
  provisioning (the booted System), so the two can no longer disagree.
- **PCIe as a selection axis (model + validate)** — host `capabilities.pcie_devices`
  becomes a list of **static descriptors** `{bdf, vendor_id, device_id, class_code, label}`
  (no `free` flag — occupancy is derived from active claims), populated by discovery. A
  request or a fleet query references a device by a **match spec** — exact `vendor:device`
  (`4hex:4hex`, a card model) or `class=NN` (`2hex` high-byte or `4hex` exact, any device of
  a type) — never by host-local BDF. A request whose device union (explicit `pcie_devices` ∪
  a shape's `pcie_match`) is non-empty makes host **selection PCIe-aware**; admission
  resolves the union to **distinct** free devices **inside the per-Resource lock**, persists
  the `(vendor:device, bdf)` snapshot claim on the Allocation, and frees it on **every
  terminal allocation transition** (release / expired / failed / break-glass) with the
  reconciler reaping orphans. A match that **no schedulable host has** is a
  `configuration_error`; a match whose only candidates are **busy** is a capacity denial that
  participates in the pending queue. No libvirt `<hostdev>` XML / IOMMU passthrough is emitted
  yet — the booking is a real reservation, the in-guest wiring is deferred to a
  provider-capability milestone (ADR-0068).
- **A reservation / work-conserving FIFO pending-queue scheduler (minimal)** —
  `allocations.request` gains an opt-in `on_capacity: "queue" | "deny"` (default `deny`,
  today's behavior). On a **capacity** denial (host cap or concurrency quota) with
  `on_capacity=queue`, the request enqueues as a `requested` allocation holding only a queue
  position — no budget reserve, no lease, **no occupancy slot** (the occupancy counters
  exclude `requested`, ADR-0069) — persisting its original inputs and leaving `resource_id`
  null; the enqueue is idempotent and bounded by `max_pending_allocations` under the PROJECT
  lock. The reconciler runs a **work-conserving** promotion sweep that **re-runs selection**
  (PCIe-aware, cordon-skipping) and moves the oldest *placeable* `requested` allocation
  `requested → granted` when a slot frees — stamping `resource_id` and running the existing
  check-then-debit + reserve + lease under the `PROJECT → RESOURCE` locks — so a free host is
  never idled behind a request waiting on a *different* busy host. Budget and configuration
  denials still hard-deny; a budget recheck failure at promotion **terminates** the request
  (not re-queues). A queued request is cancelled via `allocations.release` (no ledger credit —
  it was never reserved); one never placeable past a max-wait window is reaped to
  `failed(queue_timeout)` — a distinct category, not `lease_expired`.
- **A fleet availability view** — `resources.availability` reports, per host, the free
  capacity headroom (**`cap − count(GRANTED/ACTIVE/RELEASING)`**, the same occupancy
  predicate admission uses, so queued rows are excluded), the free PCIe devices, and the
  shapes that fit **now**; it is **schedulability-aware** (a `cordoned`/non-`available` host
  is flagged non-schedulable, never "fits now") and reports queue depth at fleet/kind
  granularity. It accepts a PCIe match filter so "which hosts have a free Intel X710 NIC?" is
  one tool call, not a per-host sweep (ADR-0070).
- **Explicit system reuse** — `systems.list` (a `viewer` view filtered by allocation, state,
  shape, and PCIe match) plus a **discover-and-validate** reuse path. `runs.create` already
  takes a `system_id` and never provisioned, so reuse adds no new attach call: it lets an
  agent attach a Run to a `ready` System it did **not** just provision (found via
  `systems.list`, possibly from another Investigation), matched by the **sizing snapshot**
  (System sizing **≥** the request + PCIe claim contains the required devices), subject to
  three locked `runs.create` preconditions — live allocation (`stale_handle`), single project
  (`configuration_error`), one non-terminal Run per System (`transport_conflict`). Reuse is
  agent-driven, never automatic (ADR-0070).

## Non-goals (scoped out)

- **No priority, preemption, or future-window booking** — the scheduler is **work-conserving**
  FIFO over the pending queue (oldest *placeable* first); a reservation cannot jump the queue
  by priority or book a slot for a stated future window. Those were the "full scheduler"
  option, declined for M1.4 (ADR-0069 §6).
- **No PCIe passthrough wiring** — M1.4 models, validates, claims, and frees devices; it
  does not emit `<hostdev>` XML, manage IOMMU groups, or bind vfio. A claimed device is a
  durable booking, not a device attached to a running guest (ADR-0068 §4).
- **No persisted `draining` resource status** — the top-level design's `ResourceStatus`
  line names `draining`, but M1.3 modeled drain as an action over the `cordoned` boolean
  (ADR-0062 §3) and the enum stays `available/degraded/offline`. A queued request simply
  never places onto a cordoned host, so a separate status earns nothing; M1.4 keeps
  `cordoned` and leaves `draining` unrealized (ADR-0069 §5).
- **No automatic / transparent reuse** — `runs.create` reuses a System only when the
  caller names one; there is no implicit match-or-provision (ADR-0070 §3).
- **No PCIe in the cost model** — kcu stays size×time on vcpus/memory (ADR-0007 §1); a
  claimed device is a capacity/availability constraint, not a billable rate input
  (ADR-0068 §5).
- **No provider-interface change** — M1.4 changes admission-core logic (the occupancy
  predicate, PCIe-aware selection, the allocation schema and state machine) and adds a
  reconciler pass, but the **provider interface is untouched**; this is not "no core change,"
  it is "no provider-seam change," and the M2 falsifiable test (adding a provider touches zero
  provider-seam lines) still holds.

## Postgres schema (M1.4 delta)

- **`system_shapes`** — a new table: `name text PRIMARY KEY`, `vcpus int NOT NULL`,
  `memory_mb int NOT NULL` (constrained to **whole-GB multiples** so the resolver maps
  `memory_mb → memory_gb` exactly, ADR-0067), `disk_gb int NOT NULL`, `pcie_match text NULL`
  (an optional match spec the shape requires, stored opaquely until the matcher lands —
  issue 4), `updated_at timestamptz NOT NULL`. Seeded by migration with `small` / `medium` /
  `large` / `max`; operator-mutable at runtime. The `shape` name on `allocations` / `systems`
  is a recorded **label, not a foreign key**, so `shapes.delete` never FK-blocks or orphans a
  live row (ADR-0067).
- **The persisted sizing identity is a snapshot of existing columns, not parallel columns**
  (ADR-0067). An Allocation already carries `requested_vcpus` / `requested_memory_gb`
  (at-grant snapshot, ADR-0007 §3) and a System already carries its `provisioning_profile`
  JSON (which holds `vcpu` / `memory_mb` / `disk_gb`). The migration therefore adds only the
  nullable **`shape` name label** to each table, plus **`requested_disk_gb`** on `allocations`
  where the existing snapshot falls short. Availability and reuse read sizing from this
  existing state, so a later `shapes.set` cannot retroactively re-size a stamped row.
- **`allocations`** further gains **`pcie_claim jsonb NOT NULL DEFAULT '[]'`** — the
  **`(vendor_id, device_id, bdf)` snapshot** of each device the allocation holds (not bare
  BDFs; the BDF is a host-local snapshot, re-resolved at provision time, ADR-0068) — and the
  queued-request changes (ADR-0069): **`resource_id` becomes nullable, guarded by a CHECK**
  (`NULL` only when `state = 'requested'`, mirroring the M1.3 `audit_log` nullable-object
  CHECK), and the **original request inputs** a queued row needs to re-admit are persisted —
  the requested **PCIe match-spec union** (to *re-resolve* at promotion), plus target/shape;
  the existing `requested_*` snapshot covers size. This is distinct from `pcie_claim`: the
  union is the *requested* specs (pre-grant), `pcie_claim` the *resolved* devices written
  only **at grant**. A partial index on `(created_at) WHERE state = 'requested'` backs the
  oldest-placeable scan.
- **`quotas`** gains **`max_pending_allocations int NOT NULL`** — a per-project cap on queued
  `requested` rows, **distinct** from the grant cap `max_concurrent_allocations` (which no
  longer counts `requested`; see the state note below) so one project cannot flood the
  backlog (ADR-0069).
- **`systems`** gains the nullable **`shape` name label** (full-custom = NULL); sizing for
  `systems.list` / reuse comes from the existing `provisioning_profile` JSON, not a new
  column.
- **Provisioning-profile schema (ADR-0024 delta)** — for a **shape-sized** allocation the
  profile's `vcpu` / `memory_mb` / `disk_gb` (today **required** `gt=0`) **become optional**:
  `systems.provision` constructs them from the resolved tuple, and a submitted profile that
  restates a *conflicting* size is `configuration_error` (ADR-0067). Issue 3 therefore spans
  the allocation selector **and** the provisioning-profile schema.
- **Resource `capabilities`** (existing jsonb) gains the **`pcie_devices`** **static
  descriptor** list `{bdf, vendor_id, device_id, class_code, label}` — **no `free` flag**
  (occupancy is derived from active `pcie_claim`s, not stored in capabilities, so a discovery
  re-scan cannot un-claim a booked device, ADR-0068). Discovery populates it like the existing
  `vcpus` / `memory_mb` / `concurrent_allocation_cap` keys.

No new state **value** is added, but the **`requested`** state changes meaning: it already
exists (`requested → granted`, ADR-0023) and was previously transited through in one
transaction (no row ever rested in it). M1.4 makes it a durable resting state for a queued
request and adds a **`requested → released`** cancellation edge. Crucially, the host-cap and
grant-quota **occupancy counters switch to a `GRANTED / ACTIVE / RELEASING` predicate** —
`requested` is removed from *occupancy* but stays **non-terminal / live** for the lease and
reconciler logic (ADR-0069); a queued row therefore holds no slot and cannot self-block its
own promotion.

## MCP tool surface (M1.4 delta)

```
Shapes catalog
  shapes.list()                                    viewer  → seeded + operator shapes
  shapes.set(name, vcpus, memory_mb, disk_gb, pcie_match?)
                                                   platform_operator  → upsert a shape
  shapes.delete(name)                              platform_operator  → remove a shape

Allocation (selector gains shape + custom + queue opt-in)
  allocations.request(selector{shape | vcpus,memory_gb,disk_gb,pcie_devices},
                      window, project, on_capacity="deny"|"queue")
                                                   operator  → granted | denied | queued(requested)
  allocations.release(allocation_id)               operator  → also cancels a queued request

Fleet availability
  resources.availability(pcie?, shape?)            viewer  → per-host headroom (occupancy
                                                              predicate), free PCIe, fitting
                                                              shapes, schedulability flag;
                                                              fleet/kind queue depth

Systems (catalog view + reuse)
  systems.list(allocation_id?, state?, shape?, pcie?)
                                                   viewer  → ready/other Systems, filterable
  runs.create(investigation_id, system_id, build_profile, …)
                                                   operator  → may target an existing ready System
```

Unchanged: `allocations.get` / `.list` / `.renew`, `accounting.estimate` / `.usage`,
`resources.list` / `.describe` / `.cordon` / `.uncordon` / `.set_status`,
`systems.provision` / `.define` / `.reprovision` / `.teardown`, the debug / control /
retrieve surfaces.

## Auth / RBAC delta

- **`shapes.set` / `shapes.delete`** gate on the M1.1 `require_platform_role`
  (`platform_operator`) — the catalog is shared fleet configuration, the same authority
  class as `ops.set_host_capacity` (ADR-0062). **`shapes.list`** is `viewer` on any
  project (a selection read, like `resources.list`).
- **`resources.availability`** and **`systems.list`** are `viewer` reads. `systems.list`
  scopes to the caller's projects (a System carries the owning attribution tuple); a
  System in an ungranted project is omitted, indistinguishable from absent (the
  no-leak rule, ADR-0023).
- **`allocations.request`** stays `operator` on the project; `on_capacity=queue` adds no
  new authority — a queued `requested` allocation is the same object the caller already
  owns, simply not yet granted.
- The reconciler promotion sweep runs under the service identity (no caller in band); the
  grant it performs writes the same `->granted` audit row `admit` writes today, attributed
  to the queued allocation's original `(principal, agent_session)` (ADR-0069 §4).

## Error taxonomy (M1.4 delta)

M1.4 adds one `ErrorCategory` and reuses existing ones for new conditions (pick the most
specific; do not invent strings):

- **`queue_timeout`** (new) — a queued `requested` allocation reaped after exceeding the
  max-wait window without ever being placeable; distinct from `lease_expired` (which means a
  *granted* lease window elapsed), so audit/SLO never conflate a never-placeable backlog
  request with a reclaimed lease (ADR-0069).
- **`configuration_error`** (reused) — an over-host shape/size, a malformed match spec, a
  PCIe match **no schedulable host has**, a cross-project reuse attempt.
- **`allocation_denied` / `quota_exceeded`** (reused) — a capacity denial; with
  `on_capacity=queue` this enqueues instead. A PCIe match whose candidates are all **busy**
  is a capacity denial (queueable), not a `configuration_error` (ADR-0068).
- **`stale_handle`** (reused) — reuse of a System whose Allocation is terminal/expiring, or a
  claimed device gone after a rescan.
- **`transport_conflict`** (reused) — reuse of a System that already has a non-terminal Run.

## Decomposition into single-PR issues

Each issue is one PR, dependency-ordered. Issues 1, 4, and 9 are foundational and
parallelizable; 8 and 10 are the leaves.

| 2 | `shapes.list` + operator `shapes.set` / `shapes.delete` | 1 | allocation |
| 3 | Shape in the allocation selector; resolved size flows into provisioning (profile-sizing conditional, ADR-0024 delta); System records its shape | 1 | allocation, provisioning |
| 4 | Host `pcie_devices` descriptor inventory + discovery + the match-spec matcher (`vendor:device` / `class`) | — | allocation |
| 5 | Custom selector `pcie_devices` → PCIe-aware selection + in-lock resolve, snapshot claim, free on every terminal transition + reconciler reap; config-vs-busy split | 4 | allocation |
| 6 | `resources.availability` — occupancy-predicate headroom + schedulability-aware + fitting shapes + PCIe match filter | 1, 4 | mcp-api |
| 7 | Pending-queue admission: durable `requested` queue state + occupancy-predicate change + nullable `resource_id`/CHECK + persisted selector + `max_pending_allocations` + `requested → released` + `on_capacity` opt-in | 3 | allocation |
| 8 | Reconciler **work-conserving** promotion sweep (re-runs selection) + `queue_timeout` reaper | 7; **5 for PCIe-aware promotion** | core-platform |
| 9 | `systems.list` view — filter by allocation / state / shape + PCIe match | — | provisioning |
| 10 | System reuse: `runs.create` attaches to a `ready` System by snapshot-≥ match, under the live-allocation / single-project / one-Run preconditions | 3, 9 | build-install |

The match-spec matcher (issue 4) is written once and consumed by issues 5 (admission
resolution), 6 (availability filter), and 9 (systems filter), so each of those PRs adds
one tool over a settled helper.

**PCIe ↔ scheduler coupling.** Because a PCIe request whose device is *busy* is a capacity
denial (ADR-0068), the queue must re-resolve PCIe at promotion: issue 7 lands the queue
**PCIe-blind first** (size/host-cap capacity only), and **PCIe-aware enqueue + promotion are
layered with/after issue 5** (the PCIe resolve/claim) — issue 8's work-conserving sweep
re-runs *PCIe-aware* selection only once issue 5 exists. An implementer must not build a
queue that cannot re-resolve a PCIe request's device on promotion.

### Sequencing & shared seams (no separate plan)

M1.4 follows the M1.3 model: detailed design lives here, and each issue is planned and
implemented end-to-end by `work-issue`. There is **no `docs/plans/m1.4-implementation.md`**.
The cross-issue concerns that no single issue can own are pinned here so parallel PRs do not
collide:

- **Migrations are one-file-per-issue, numbered, claimed up front.** Schema-touching issues
  add a new forward-only `NNNN_*.sql` (next free is `0013`); two parallel PRs would otherwise
  both grab the same number. **Pre-assign**: 1 = `system_shapes`, 4 = `pcie_devices` (jsonb,
  no DDL — capabilities is existing), 7 = the queue/occupancy columns + `quotas.max_pending`
  + the ADR-0024 profile-sizing delta. The second schema PR to merge renumbers its file on
  rebase; `tests/db/test_migrate.py` gates the chain, so renumber-on-rebase is expected.
- **Land the shared seams before their consumers.** Issue **4** (the match-spec matcher) must
  merge before 5/6/9 (which consume it). Issue **7** (the `GRANTED/ACTIVE/RELEASING`
  occupancy predicate + `requested` queue state) must merge before **6** (availability reads
  the same predicate) and **8**. So the merge wave is: **1, 4 first** (independent), then
  **7**, then the fan-out **2/3/5/6/9** in parallel, then leaves **8, 10**.
- **Expected rebase zones** (recurring in M1.3 parallel work): `app.py` tool registration, the
  generated `docs/guide/reference/*` (regenerated by `docs-check` whenever a tool is added —
  rebase, don't hand-edit), `tests/mcp/core/test_tool_docs.py`, and `tests/db/test_migrate.py`.
  Treat these as merge-time rebases, not design conflicts.
- **Each issue runs in its own worktree** (parallel `work-issue` requires it, per the global
  standard) — never share the working copy.

## Carried invariants

1. **Fail-closed admission** (ADR-0007 §2) — a shape that resolves to an over-host size,
   an unresolvable PCIe match, or a malformed selector is a `configuration_error` with no
   durable write, exactly as a raw oversized selector is today.
2. **Budget is charged at grant, never at enqueue** (ADR-0069 §2) — a `requested`
   allocation holds a queue position only; the `reserved` ledger debit and the lease
   window are written when the promotion sweep grants it, so a request that waits and is
   later canceled costs nothing.
3. **A device claim is honest** (ADR-0068) — a device is not-claimable by another allocation
   for the life of this allocation (occupancy derived from active `pcie_claim`s, not a stored
   `free` flag) and freed on **every terminal transition** (release / expired / failed /
   break-glass) with the reconciler reaping orphans, so availability counts and the admission
   matcher never double-book a device, even though no guest is attached to it.
