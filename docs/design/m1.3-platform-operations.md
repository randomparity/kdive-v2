# M1.3 — Platform Operations (Integration Contract)

## Purpose

M1.3 builds the **mutating platform-operations surface** that M1.1 deferred. M1.1 settled the
platform-role model (`platform_admin` / `platform_operator` / `platform_auditor`), the
`require_platform_role` seam, and the `platform_audit_log` read-access table, but shipped only
one consumer (`accounting.report`). [ADR-0043](../adr/0043-platform-scoped-rbac-tier.md) §5
deferred every *mutating* platform tool and the second/third auditor reads to M1.3 so they
could be shaped against operational use. M1.3 ships them. **Because the authorization machinery
already exists, M1.3 adds tools and one hardening retrofit — not authorization plumbing.**

## What M1.3 adds to M1.1

- **`platform_operator` infra tools** — `resources.set_status`, `resources.cordon` /
  `uncordon`, `resources.drain` (mode-dependent role: `mode=passive` is `platform_operator`,
  `mode=force_release` escalates to `platform_admin`); `ops.reconcile_now`; `ops.queue_pause` /
  `queue_resume` / `jobs_list`; `ops.set_cost_class_coeff` / `set_host_capacity`.
- **`platform_admin` break-glass** — `ops.force_teardown`, `ops.force_release` — a separate
  path that fully overrides the three-check destructive gate, with a mandatory `reason` and an
  always-written `platform_audit_log` row.
- **The remaining `platform_auditor` reads** — `audit.query` (project `admin` form +
  cross-project `platform_auditor` form) and `inventory.list` (cross-project systems/allocations
  summary).
- **The bare-`require_role` denial-audit retrofit** — a new `RoleDenied` exception (introduced
  this milestone; today `require_role` raises the bare `AuthorizationError`) raised at the
  **member-rank-below** site only and caught at the dispatch boundary, writing the reserved bare
  `transition='denied'` literal to `audit_log` via a guard-exempt `audit.record_denial()`, for
  member-over-reach denials only. `RoleDenied` carries the `project` (the boundary cannot recover
  it from call args for object-resolving tools), which keeps `audit_log.project` `NOT NULL`. The
  non-member site keeps the base `AuthorizationError` (so it is never caught, never audited). The
  destructive gate's existing `{op}:denied` rows always carry an object and so satisfy the same
  CHECK without exemption (ADR-0062 §5).

## Non-goals (scoped out)

- **No job requeue/force-fail or worker orchestration** — queue control is pause/resume +
  cross-project inspect only; the reconciler already auto-repairs abandoned jobs (ADR-0062 §1).
- **No runtime `W_CPU`/`W_MEM` weight tuning** — the global reference weights stay code-config;
  only cost-class coefficients and per-host capacity caps are runtime knobs.
- **No per-project budget/quota change** — those stay on `accounting.set_budget` /
  `accounting.set_quota` (project `admin`).
- **No persisted `draining` status** — drain is an action over the `cordoned` boolean (ADR-0062 §3).
- **No control-plane freeze from `queue_pause`** — pause stops the worker's claim loop only; the
  reconciler keeps running and may enqueue jobs that wait for resume (ADR-0062 Consequences).
- **No new role model** — `platform_roles`, `require_platform_role`, `platform_audit_log` are
  M1.1; M1.3 only consumes them. No project/role management (the IdP issues claims).
- **No core/plane interface change** — M1.3 is tools over settled seams.

## Postgres schema (M1.3 delta)

- **`resources`** gains a boolean **`cordoned`** column (`NOT NULL DEFAULT false`) — schedulability,
  orthogonal to the health `status` enum (a host can be `cordoned` *and* `degraded`/`offline`;
  the columns never clobber).
- **`ops_control`** — a new single-row table: `queue_paused boolean NOT NULL DEFAULT false`
  (with `updated_at` and a single-row guard). The worker reads it before each `dequeue`.
- **`audit_log`** — `object_kind` / `object_id` become **nullable** (`project` stays `NOT NULL`),
  guarded by a CHECK keyed on `transition = 'denied'` (object columns may be NULL only on a
  `'denied'` row). The denial-audit row uses the reserved bare `'denied'` literal; the destructive
  gate's `{op}:denied` rows always carry an object and satisfy the CHECK's object-present branch, so
  both denial kinds coexist under the one CHECK. A `reason text` column is added (set by break-glass
  and denial rows, NULL elsewhere).

No new table for pricing/capacity: cost-class coefficients (`cost_class_coefficients`) and host
caps (resource `capabilities` jsonb) are already DB-backed.

## MCP tool surface (M1.3 delta)

```
Resources (host ops — platform_operator, except drain mode=force_release → platform_admin)
  resources.set_status(resource_id, status)         health: status ∈ {available, degraded, offline}
  resources.cordon(resource_id)                      cordoned := true (placement skips)
  resources.uncordon(resource_id)                    cordoned := false
  resources.drain(resource_id, mode=passive)         platform_operator: cordon + report live allocations
  resources.drain(resource_id, mode=force_release, reason)
                                                     platform_admin: cordon + force-release each via the
                                                     ops.force_release break-glass path; returns a
                                                     per-allocation result list (released/failed/skipped)

Ops (control plane — platform_operator)
  ops.reconcile_now()                                one reconcile_once() pass; returns summary
  ops.queue_pause() / ops.queue_resume()             toggle queue_paused
  ops.jobs_list(filters…)                            cross-project queue depth / job states
  ops.set_cost_class_coeff(cost_class, coeff)        upsert pricing coefficient
  ops.set_host_capacity(resource_id, concurrent_allocation_cap)

Ops (break-glass — platform_admin)
  ops.force_teardown(system_id, reason)              bypasses the three-check gate; always audited
  ops.force_release(allocation_id, reason)           bypasses the three-check gate; always audited

Audit / inventory (reads)
  audit.query(filters…)   project form  → require_role(project, admin)
                          x-project form → platform_auditor [read-audited]
  inventory.list(filters…)               → platform_auditor [read-audited]
```

Unchanged: per-project `systems.teardown` / `allocations.release` / `jobs.*` / `resources.list`
/ `.describe` / `accounting.set_budget` / `.set_quota`.

## Auth / RBAC delta

All new tools gate on the M1.1 `require_platform_role` seam: `platform_operator` for infra and
control-plane ops, `platform_admin` for break-glass, `platform_auditor` for the cross-project
reads (satisfied by `platform_admin` per ADR-0043 §2). **`resources.drain` is mode-dependent**:
`mode=passive` (cordon + report) is `platform_operator`, but `mode=force_release` is a
cross-project destructive action and escalates to `platform_admin` + a mandatory `reason`, routing
each release through the same break-glass attribution path as `ops.force_release` — an operator can
cordon and passively drain, only an admin can forcibly evict tenants. `audit.query`'s
project-scoped form rides `require_role(project, admin)`. Break-glass **fully overrides** the
three-check destructive gate (authority = `platform_admin` + mandatory `reason` + audit row written
through the guard-exempt attribution writer, since the admin is not a member of the target project).
Cross-tenant actions record to `platform_audit_log`. `require_role` **member-over-reach** denials
record to `audit_log` (reserved bare `transition='denied'`) via a `RoleDenied` catch — `RoleDenied`
is raised at the rank-below site only and carries the `project`; non-membership denials (base
`AuthorizationError`) and `require_platform_role`/`DestructiveOpDenied` are excluded (ADR-0062 §5).
The deployment root remains out-of-band (ADR-0043 §7).

## Exit criteria

Falsifiable; each becomes a test. (Full statements in the detailed design §Exit criteria.)

1. **Cordon** excludes a host from placement on **both** selection paths without touching its
   existing allocations: the pick-by-kind query skips a `cordoned`/non-`available` host, **and** an
   explicit `resource_id` naming a cordoned host is rejected (naming the host by id is not a
   placement escape hatch). `uncordon` restores it; cordon and `status` are independent columns (a
   host can be `cordoned` *and* `degraded`; neither verb clobbers the other's column).
2. **Drain** `mode=passive` (`platform_operator`) leaves live allocations running; a
   `mode=force_release` call from a `platform_operator` is denied (it requires `platform_admin`);
   a `platform_admin` `force_release` empties the host and returns a per-allocation result list
   (released/failed/skipped) so a partial drain is observable and re-invokable; both modes leave
   the host `cordoned`.
3. **`reconcile_now`** resolves a pending repair in one call and returns a summary, leaving the
   periodic loop unaffected; it runs the same advisory-locked `reconcile_once()` pass as the
   periodic reconciler, so an on-demand pass and a concurrent periodic pass cannot double-act.
4. **Queue pause** stops new claims but not a job already in flight; `resume` restores claiming.
5. **Tuning** — `set_cost_class_coeff` changes the next charge (not committed ledger rows);
   `set_host_capacity` changes admission's cap without evicting.
6. **Break-glass** succeeds against a cross-project object that fails the three-check gate for a
   non-member `platform_admin`; the audit write succeeds despite non-membership (guard-exempt
   attribution); a blank `reason` is rejected; every call writes a `platform_audit_log` row; a
   `platform_operator` token is denied.
7. **Auditor reads** — `audit.query` project form requires `admin` on that project; its
   cross-project form and `inventory.list` require `platform_auditor`, are denied to a
   project-only token, and write a `platform_audit_log` row (not an `audit_log` row).
8. **Denial-audit** — a **member-over-reach** `require_role` denial writes one `audit_log` row
   with `transition='denied'` (principal/agent_session/tool/**project**, object NULL); a
   **non-membership** denial writes no row; `require_platform_role`/`DestructiveOpDenied` are not
   caught by the `RoleDenied` boundary; the CHECK rejects a non-`denied` row with a NULL object;
   the success path is unchanged.
