# M1.1 — Platform-Scoped RBAC Tier (Integration Contract)

## Purpose

M1.1 adds the **platform-scoped authorization tier** that the per-project model
cannot express. M0/M1 roles are strictly per-project (`roles: {project: role}`,
`require_role(ctx, project, role)`); there is no actor that can act across projects
or on shared infrastructure — the cross-project role
[ADR-0006](../adr/0006-oidc-rbac-attribution.md) explicitly deferred as "a separate
explicitly-granted claim." M1.1 introduces that claim and ships the **seam** plus the
one cross-project read with a live consumer, **`accounting.report`**. It is the first
entry in the M1.x local-libvirt feature-deepening band and is foundational to the
M1.2 live-stack validation (which exercises the platform claim over the wire) and to
M1.3 platform operations.

## What M1.1 adds to M1

- A new token claim **`platform_roles`** (flat array), parsed fail-closed by
  `platform_roles_from_claims()`, separate from the per-project `roles` map.
- A **`PlatformRole`** enum — `platform_admin`, `platform_operator`,
  `platform_auditor` — granted **independently** (not ranked), with one partial-order
  exception: `platform_admin` satisfies `platform_auditor`; `platform_operator`
  satisfies neither.
- `RequestContext.platform_roles: frozenset[PlatformRole]` and
  `require_platform_role(ctx, role)`, the sole enforcement seam **for platform roles**,
  orthogonal to `require_role` (which still guards per-project authority, including
  `accounting.report`'s granted-set form).
- A new **`platform_audit_log`** table + `audit.record_platform()` writer (no
  project-membership guard) for read-access logging of platform reads,
  **multi-project member reads** (a granted-set rollup spanning >1 project or using
  `group_by=principal`, even with no platform role), and platform-role denials **by a
  principal holding ≥1 `platform_role`** (a no-platform-role caller's denial is the routine
  non-grant case and is not recorded — ADR-0043 §4) — the
  per-project `audit_log` cannot represent a project-independent action (its
  `project`/`object_*` are `NOT NULL` and `audit.record()` requires `project in ctx.projects`).
- One read tool: **`accounting.report`** — a multi-project usage/billing rollup +
  per-allocation reserved/reconciled variance, with optional group-by-principal /
  time-window dimensions, in **two scope forms** (ADR-0043 §3): a **granted-set** form
  over the caller's own member projects (authorized per project by `require_role(…, viewer)`,
  **no platform role**; audited only when it spans >1 project or uses `group_by=principal` —
  the manager/per-team-FinOps path) and an **all-projects** form gated `platform_auditor` and
  always read-audited (the oversight path the live-stack driver exercises). The granted-set
  form is **P1-independent** — it rides `require_role`, not the platform seam — and is
  co-delivered only because it shares the `accounting.report` tool with the all-projects form;
  its place in M1.1 is packaging, not an authorization dependency.

## Non-goals (deferred to M1.3 / out of scope)

- **No infra-mutation or break-glass tooling** — host cordon/drain/maintenance,
  reconcile/worker control, capacity/cost tuning, cross-project teardown/force-release
  are M1.3 (Platform operations).
- **No `audit.query` or `inventory.list`** — the other platform-auditor reads park in
  M1.3 with the operator tooling (their consumers pair with it).
- **No full bare-`require_role` denial-audit retrofit** — M1.3.
- **No project or role management** — `roles`/`platform_roles` are issued by the IdP;
  kdive only reads and enforces them ([ADR-0043](../adr/0043-platform-scoped-rbac-tier.md) §6).
- **No change to per-project RBAC** — `roles`, `roles_from_claims`, `require_role` are
  untouched; `accounting.usage` stays `viewer`.

## Postgres schema (M1.1 delta — migration `platform_audit_log`)

A new append-only table `platform_audit_log(ts, principal, agent_session,
platform_role, tool, scope, args_digest)` — `scope` describes the breadth of the read
(e.g. `all-projects` or the project set) rather than a single `project`/`object_id`.
`platform_role` is **nullable**: it records the role for a `platform_auditor`/`platform_admin`
read and is null for an audited granted-set member read (which carries no platform role).
The per-project `audit_log` is unchanged. `accounting.report`'s group-by-principal needs
**no schema change**: `principal` is obtained by joining `ledger.allocation_id →
allocations.principal` (the existing `ledger_allocation_id_idx` covers the join).

## MCP tool surface (M1.1 delta)

```
Accounting (report)
  accounting.report(scope, window, group_by, …)
      scope=granted-set  → rollup over the caller's member projects (ctx.projects)
                           [require_role(p, viewer) per project; no platform role;
                            audited → platform_audit_log iff >1 project or group_by=principal]
      scope=all-projects → rollup over every project
                           [platform_auditor; always read audited → platform_audit_log]
      both forms: per-allocation reserved/reconciled variance; optional
      group_by=principal (ledger ⋈ allocations) and/or time window
```

Unchanged: `accounting.usage` / `.estimate` / `.set_budget` / `.set_quota` (per-project).

## Auth / RBAC delta

The platform tier is an **independent scope axis** from per-project roles (ADR-0043 §7:
tenant-data scoped by `project`; infra scoped by location/zone). `platform_admin`
implies `platform_auditor`; the two scopes never interact. `accounting.report`'s
**granted-set** form needs **no platform role** — it is authorized per project by
`require_role(ctx, p, viewer)` over the caller's `ctx.projects` (a named non-member project is
rejected, not dropped; a zero-project resolution returns an empty rollup), so a manager sees
their own teams without an all-tenant grant; only the **all-projects** form requires
`platform_auditor`. The **deployment root** (service operator) remains out-of-band — not an
RBAC role.

## Exit criteria

Falsifiable; each becomes a test.

1. **Claim parsing is fail-closed.** `platform_roles_from_claims` accepts a valid
   array, returns `{}` for an absent claim, and raises `AuthError` for an unknown role
   or a non-array claim.
2. **`require_platform_role` enforces the partial order.** Each role satisfies itself;
   `platform_admin` satisfies `platform_auditor`; `platform_operator` satisfies neither;
   a project-only token (empty `platform_roles`) is denied.
3. **`platform_audit_log` exists and is writable without project membership.** The
   migration creates the table; `audit.record_platform()` writes a row for a principal
   with **empty `ctx.projects`** and for a `require_platform_role` denial.
4. **`accounting.report` all-projects form is correct, gated, and audited.** A
   `platform_auditor` token gets an all-projects rollup spanning **≥2 projects** with
   per-project and total variance matching a hand-computed ledger; a project-only
   `viewer`/`operator`/`admin` token is **denied the all-projects form**; each successful
   all-projects call writes a `platform_audit_log` row (not an `audit_log` row).
5. **`accounting.report` granted-set form respects membership and is audited by read-shape.**
   A token that is a member of projects A and B (no platform role) gets a rollup over exactly A
   and B (defaulting to `ctx.projects`); explicitly **naming** a project C the caller does not
   belong to is **rejected** (`authorization_denied`), not silently dropped; a request
   resolving to **zero** member projects (empty `ctx.projects`) returns an **empty rollup, not
   an error**. A multi-project read (>1 project) or one using `group_by=principal` writes a
   `platform_audit_log` row (with `platform_role` null); a single-project rollup without
   principal grouping writes **no** audit row (matching `accounting.usage`). Neither writes a
   per-project `audit_log` row.
6. **Group-by-principal / time-window is correct.** With `group_by=principal` over a window,
   the report returns per-principal totals computed via `ledger ⋈ allocations` (on
   `allocation_id`) matching a hand-computed expectation, in both scope forms.
