# Plan — #185 M1.5 (6): admission-control concurrency races

A **test-only** PR (`type:test`). Contend the ADR-0040 admission locks for real against a
**fault-inject** Resource and assert, at the cap boundary, the three invariants the spec
§Validation surface names: **exactly one grants**, **no budget overspend** (the
check-then-debit is atomic), **no capacity double-book**. No production behavior changes —
this exercises `services/allocation_admission.py` as it already stands (carried invariant 4).

## What already exists (and is NOT enough)

`tests/adversarial/test_admission_concurrency.py` already races real connections through
`admit()` and proves the **per-resource capacity** lock never overshoots the host cap. But:

- It seeds a **`local-libvirt`** Resource, not the fault-inject kind #185 names.
- Budget/quota are seeded **generous**, so it never proves the **budget** check-then-debit is
  atomic under contention — a `spent_kcu` overspend race is entirely untested.
- It relies on the advisory lock to serialize the racers but never **forces** the racers to
  overlap inside the locked section, so a regression that shrinks the lock's scope could pass
  by luck (the critical section is fast enough that racers rarely overlap).

#185 fills exactly those three gaps. It does **not** duplicate the host-cap-overshoot cases
already covered; it adds the budget-binding race and the fault-inject target.

## The "injected latency / slow release" seam (no production feature)

The spec says "the mock keeps allocations active (injected latency / slow release) so
contention is real, not a window race." In this codebase the deterministic, test-side way to
do that is the existing `tests/adversarial/test_idempotency_concurrency.py` pattern: an
`asyncio.Barrier` + `asyncio.sleep` injected via **monkeypatch** into a function the
critical section calls, so every racer is provably in-flight before any one commits. This
adds **no** production hook/flag (global standard: no speculative features) — the latency
lives in the test, mirroring "the mock keeps allocations active."

- **Budget race:** monkeypatch `kdive.services.allocation_admission.within_budget` (the
  re-export the gate calls) to `await barrier.wait(); await asyncio.sleep(d)` *before*
  delegating to the real check. This forces all racers to read `spent_kcu` while every other
  racer is also inside the PROJECT-locked section — the worst case for an overspend. If the
  PROJECT lock did NOT serialize check-then-debit, two racers would both see remaining ≥
  estimate and both debit, overspending; with the lock, only the budget-affordable count
  grants and `spent_kcu` never exceeds `limit_kcu`.
- **Capacity race:** the existing suite already barrier-free-proves this; #185 adds **one**
  barrier-forced capacity case against the fault-inject resource so the double-book invariant
  is pinned with a guaranteed overlap (not luck), parallel to the budget case.

The barrier must be sized to the number of racers; with the PROJECT lock held, only ONE racer
can be inside the locked section at a time, so a barrier sized to the racer count would
**deadlock** (the holder waits for peers that are blocked on the advisory lock it holds). This
is the load-bearing subtlety: **the monkeypatched delay must be a plain `asyncio.sleep`, not a
barrier rendezvous, for the budget/capacity cases** — the advisory lock already serializes, so
a per-call sleep is what widens the held-lock window and makes a non-atomic check-then-debit
lose deterministically. A barrier sized to the racer count belongs only to a **lock-free**
control test (below) that proves the sleep actually exposes the race when the lock is absent.

## Tests (TDD, all in `tests/adversarial/test_admission_fault_inject_races.py`)

New conftest helpers (in `tests/adversarial/conftest.py`):
- `seed_fault_inject_resource(conn, *, cap)` — a `ResourceKind.FAULT_INJECT` Resource with the
  concurrent-alloc cap + generous vcpus/memory caps + a minimal fault-inject `capabilities`
  seed (seed/fault_rate/max_latency_s) so it is a well-formed bookable fault-inject row. Mirror
  `seed_resource` but with the fault-inject kind.
- `seed_budget` already takes `limit`; reuse it to seed a **tight** budget.

1. `test_concurrent_request_no_budget_overspend` — fault-inject resource, host cap generous,
   budget tight: `limit_kcu` admits **exactly B** of the small selectors (B < racers). Inject
   the per-call sleep into `within_budget`. Assert: grants == B; denials == racers − B with
   category `ALLOCATION_DENIED`; final `spent_kcu == B * estimate` and `spent_kcu <= limit_kcu`
   (no overspend); `count(allocations) == B`; one audit row per grant.
2. `test_concurrent_request_budget_exact_boundary` — `limit_kcu` exactly == racers * estimate:
   every racer must grant, none spuriously denied, `spent_kcu == limit_kcu` exactly (the
   boundary is inclusive, `remaining >= estimate`).
3. `test_concurrent_request_no_capacity_double_book` — fault-inject resource, budget generous,
   host cap K < racers, per-call sleep injected into the capacity count: grants == K, denials
   carry `reason="at_capacity"`, `count(allocations) == K`. (The fault-inject parallel to the
   existing local-libvirt capacity case, with a guaranteed-overlap sleep.)
4. `test_concurrent_request_both_locks_contended` — BOTH bind: cap K and budget that affords B,
   with K != B; the binding minimum `min(K, B)` grants, `spent_kcu <= limit_kcu`,
   `count(occupying) <= K`. Proves the PROJECT→RESOURCE lock order holds both invariants at
   once with no deadlock.
5. `test_sleep_in_unlocked_check_would_overspend` (control / falsification) — drive the **bare**
   `within_budget` + a manual non-atomic check-then-debit on distinct connections **without**
   the PROJECT lock, barrier-forced overlap, and assert it DOES overspend. This proves the
   sleep injection genuinely exposes the race, so test 1's pass is the lock's doing, not a
   too-fast critical section. (Mirrors `test_idempotency_concurrency`'s bare-vs-locked pair.)

Edge/error coverage: `racers == B` and `racers == K` exact boundaries (no spurious denial);
budget-binding denial category is `ALLOCATION_DENIED` (not `QUOTA_EXCEEDED` — quota is
generous); cap-binding denial reason is `at_capacity`.

## Verification

- `just test` runs the new module (the adversarial suite is unmarked, runs under the default
  gate); the disposable-Postgres fixtures (`migrated_url`) back it. `KDIVE_REQUIRE_DOCKER=1`
  in CI turns a missing Docker daemon into a hard failure, so the race tests actually run in CI.
- `just lint`, `just type` (whole-tree), `just ci` green before every commit.
- Mutation-style confidence: test 5 (lock-free control) is the falsification — break the
  PROJECT lock (conceptually) and test 1 fails; that pairing is the regression proof.
