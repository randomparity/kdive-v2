# Plan — #162 custom selector `pcie_devices`: resolve / persist / release the claim

ADR-0068, spec m1.4-system-catalog-scheduling.md. Builds on the merged matcher
(`kdive.domain.pcie`, #158). Greenfield rewrite tool surface (`allocations.*`).

## Goal

Make PCIe a requestable, durably-claimed selection axis:

1. `allocations.request` selector gains `pcie_devices: [match-spec, ...]`.
2. A non-empty device union makes host **selection PCIe-aware** (pick a schedulable
   host with a free matching device for every spec).
3. Admission resolves the union to **distinct** free devices **inside the per-Resource
   lock** and persists `allocations.pcie_claim` (the `(vendor_id, device_id, bdf)`
   snapshot).
4. The claim frees on **every terminal transition** (occupancy is derived from
   non-terminal allocations' claims — no explicit clear needed) + reconciler reaps.
5. config-vs-capacity split; kcu unchanged by a claim.

## Key architectural facts (verified by reading the tree)

- Occupancy is **derived**: `pcie.resolve_*` subtracts BDFs held by **active** claims.
  So "free the claim" = drive the allocation terminal. Every terminal path already
  does that (`allocation_release._release_locked`, reconciler `_expire_one`,
  break-glass via the same release mechanic, failure via state transition). **No new
  free-the-claim code is required** — the claim row persists as a historical snapshot
  and simply stops counting once the allocation is terminal. The reconciler's existing
  expiry sweep + orphan reaping covers orphans. Tests assert the *behaviour* (a freed
  device is re-claimable), not a column write.
- The per-Resource lock is taken in `allocation_admission._admit_under_project_lock`
  (`advisory_xact_lock(RESOURCE, resource.id)`), wrapping `_host_cap_check` + `_grant`.
  The PCIe resolve-and-claim MUST go here (ADR-0068 Consequences): a locked
  read-modify-write, NOT pre-lock validation.
- Pre-lock: only grammar validation (`parse_match_spec`) — fail-closed
  `configuration_error`, no durable write.
- Host selection today: MCP `request_allocation` calls `_resolve_resource` (PCIe-blind,
  picks first schedulable host of a kind, or validates a by-id host). PCIe-aware
  selection extends this for the non-empty-union case.

## Selection vs. claim semantics (challenge finding 1 — resolved)

Selection (pre-lock, possibly across candidate hosts) is a **best-effort filter**; the
in-lock resolve-and-claim under the chosen host's RESOURCE lock is **authoritative**.
Once selection picks a host, admission claims on that host or denies — it does NOT
re-select a different host on an in-lock CAPACITY. Consequence, stated honestly: two
concurrent by-kind PCIe requests for the last card on host A, where host B also has a
free card, can both land on A; one wins, the other gets a **CAPACITY denial** instead of
being rerouted to B. This is acceptable for M1.4 because a CAPACITY denial is exactly the
queueable case the #164 work-conserving promotion sweep re-resolves (ADR-0068 §"Two
distinct failure modes"; spec §"PCIe ↔ scheduler coupling"). Fleet-wide re-selection is
#164's job, not #162's. A test documents this (one wins, one CAPACITY), so the limitation
is pinned, not implicit.

The occupancy read in selection holds no per-host RESOURCE lock (a dirty read by design);
correctness rests entirely on the in-lock re-resolve. Never treat selection's free/busy
view as authoritative.

### Resolution / selection (domain + admission)
- `domain/pcie.py`: add `resolve_union_specs(specs, descriptors, *, claims)` thin alias
  over `resolve_multiset` so the call site reads intent (the union). Keep matcher core
  unchanged. (If trivial, fold directly — avoid premature abstraction. Decision at
  impl: use `resolve_multiset` directly; add a `descriptors_from_capabilities` +
  `active_claims_for_resource` helper instead, which is the real new logic.)
- New `services/pcie_claim.py` (admission helpers):
  - `requested_specs(selector_payload) -> list[str]` — the explicit `pcie_devices`
    list (shape `pcie_match` union composes here later: `explicit + shape_specs`).
  - `descriptors_for(resource) -> list[PCIeDescriptor]` — read+validate
    `capabilities.pcie_devices`.
  - `active_claims(conn, resource_id) -> list[PCIeClaim]` — union of `pcie_claim`
    across the host's non-terminal allocations (the occupancy set). MUST reuse the
    existing `_NON_TERMINAL` state set from `allocation_admission` (export it) so the
    occupancy predicate can never drift from the capacity predicate
    (challenge finding 2). A RELEASING allocation's claim still counts as occupied —
    explicit test.
  - `resolve_and_claim(conn, resource, specs) -> ClaimResolution` (MATCHED+devices /
    CONFIG / CAPACITY) — called under the RESOURCE lock.
  - `select_pcie_host(conn, candidates, specs) -> SelectionResult` — PCIe-aware host
    pick: first schedulable host whose multiset fully resolves; else CONFIG (no host's
    descriptors match any spec) vs CAPACITY (matched but busy).
- `allocation_admission.AllocationRequest`: add `pcie_specs: tuple[str, ...] = ()`.
  In `_admit_under_project_lock`, after the host-cap check passes and before `_grant`,
  call `resolve_and_claim`; thread the resolved devices into `_grant` to persist
  `pcie_claim`. A CONFIG/CAPACITY outcome → typed denial, no write (txn rolls back).
  AdmissionOutcome already carries `category`; map CONFIG→configuration_error,
  CAPACITY→allocation_denied (most specific existing category; queueable later #164).
- MCP `request_allocation`: parse `pcie_devices` from the payload, validate grammar
  pre-lock (config_error on malformed), do PCIe-aware host selection for the non-empty
  case, thread specs into the domain request.
- `AllocationRequestPayload`: add `pcie_devices: list[str] = []`.

### kcu invariance
- `pcie_specs` are NOT passed to `cost/rate/estimate`. A test asserts the estimate is
  identical with and without a claim.

## Terminal-free coverage (for #164/#165 reviewers)
- release: `allocation_release._release_locked` → `released`.
- expiry: `reconciler._expire_one` → `expired`.
- failure: allocation → `failed` is a legal state-machine edge (state.py) but NO service
  drives an allocation there today (System/Run failures, not allocation). So the
  "failure frees the claim" test is a **state-machine assertion** — a `failed`
  allocation's claim is excluded by the occupancy predicate — not a lifecycle-trigger
  test (challenge finding 4). Written as: seed a `failed` allocation holding a claim,
  assert its device is re-claimable.
- break-glass: `breakglass_release_allocation` → `release_with_backstops` → `released`.
- orphan reap: reconciler expiry sweep moves a non-terminal orphan terminal.
All free the claim because occupancy = non-terminal allocations' claims.

## Risks
- Concurrency: the no-double-book test must drive two admits racing the last free
  device through the RESOURCE lock (mirror `test_admit_blocks_behind_a_held_resource_lock`).
- Descriptor trust: `label` is untrusted libvirt text; `pcie_claim` snapshots only
  `(vendor_id, device_id, bdf)` (no label) — already the matcher's `PCIeClaim` shape.
- Config-vs-capacity split is tested **explicit-only** (correct for #162). The matcher's
  CONFIG-dominates-CAPACITY rule interacting with a future shape-spec union member is
  #161's responsibility — not validated here (challenge finding 3).
- No new MCP tool (modifies existing `allocations.request`) → test_tool_docs likely
  unchanged; regenerate generated docs if the recipe touches them.
