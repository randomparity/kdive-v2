# Plan — #164 M1.4 (7): pending-queue admission

Make `requested` a durable queue state for capacity-denied requests (ADR-0069), PCIe-blind.

## Migration `0016_pending_queue.sql`
- `quotas.max_pending_allocations integer NOT NULL DEFAULT 0` — distinct from the grant cap.
  Backfills existing rows to 0 (queue is opt-in; a project must be granted pending depth
  explicitly via `set_quota`, fail-closed like the other quota caps). New column is set by
  `set_quota`. **Consequence (finding 5):** `on_capacity=queue` is dead-on-arrival for a
  default-seeded project until an operator raises `max_pending_allocations`. Default stays 0
  (fail-closed, mirrors the other caps; no silent non-zero default). Therefore EVERY enqueue
  test/fixture must set it `> 0` explicitly, and #165's promotion demo/runbook must raise it.
- `allocations.resource_id` → **nullable**, guarded by CHECK
  `allocations_resource_id_state_check`: `resource_id IS NOT NULL OR state = 'requested'`
  (NULL only legal in `requested`). Mirrors the 0012 audit_log nullable-object CHECK.
- Persisted request inputs to re-admit (distinct from `pcie_claim`, which is resolved-at-grant):
  - `requested_pcie_specs jsonb NOT NULL DEFAULT '[]'` — the requested match-spec **union**.
  - `requested_kind text` — the by-kind target (NULL when targeted by id).
  - (the by-id target is `resource_id` itself when present; a queued by-id request stamps
    `resource_id` is NULL though — so we also need to remember the requested id.)
  - `requested_resource_id uuid` — the by-id target persisted for re-admit (NULL when by-kind).
  Size already covered by `requested_vcpus`/`requested_memory_gb`/`requested_disk_gb`; `shape`
  exists. No parallel sizing columns.
- Partial index `idx_allocations_requested_created_at ON allocations (created_at)
  WHERE state = 'requested'` — backs #165's oldest-placeable scan.
- Update BOTH applied-version lists in `tests/db/test_migrate.py` (two "0015" lists → add "0016").

## Enqueue (admission)
- `AllocationRequest` gains `on_capacity: Literal["deny","queue"] = "deny"`,
  `requested_kind`/`requested_resource_id` (the target descriptor for persistence).
- **Queueable set (finding 1) = host-cap (`at_capacity`) + grant-quota (`quota_exceeded`)
  ONLY.** A BUDGET denial returns `ALLOCATION_DENIED` with `reason=None` — the SAME category
  as the host-cap denial (which carries `reason="at_capacity"`) — so the discriminator MUST
  NOT branch on `category`. Carry an explicit `queueable: bool` on each `AdmissionOutcome`
  denial: set it `True` on the grant-quota and host-cap denials, `False` on budget,
  configuration, and PCIe denials. Budget and configuration denials **stay hard-deny**.
- **PCIe-CAPACITY hard-denies in #164 (finding 2).** A PCIe-busy request never reaches
  `admit`: `_select_target` returns `allocation_denied` with `resource=None` before `admit`
  is called (`allocations.py:138-143,227-228`). So there is no chosen host to persist. The
  in-lock `_resolve_pcie_claim` CAPACITY denial keeps `queueable=False`. Queued PCIe is #165.
  This keeps #164 genuinely PCIe-blind.
- Enqueue flow: when `on_capacity=queue` AND the denial is `queueable`, `admit` enqueues
  instead of returning the denial. **All of the following run in the ONE PROJECT-locked
  transaction `admit` already opens (finding 3), mirroring `_grant`:**
  1. pending-cap check: `SELECT count(*) ... WHERE project = %s AND state = 'requested'`
     (literally `state = 'requested'`, NOT the occupancy predicate) `< max_pending_allocations`
     (no quota row → 0 cap → over; `quota_exceeded`, no write).
  2. insert `requested` Allocation: `resource_id=None`, `lease_expiry=None`, no reserve, empty
     `pcie_claim`, persisted `requested_*`/`shape`/`requested_pcie_specs`/target.
  3. record idempotency key (same `_REQUEST_KIND`) so a replay returns the queued row.
  4. audit `->requested`.
- The host-cap denial is detected inside the nested RESOURCE lock; the quota denial before it.
  Enqueue runs under the PROJECT lock either way (the cap-count + insert are atomic; the
  nested RESOURCE lock is released first or held — either is safe since the cap is project-
  scoped, not resource-scoped). Simplest: route to a single `_enqueue` helper from both denial
  sites, all inside the PROJECT-locked transaction.
- Idempotency: the existing top-of-lock `resolve_replay` already returns the queued row on
  replay (it returns whatever allocation the key recorded), so a replayed enqueue is a no-op.

## Release (cancellation)
- `allocation_release.py` (finding 4): **do NOT add `requested` to `_RELEASABLE`** — that
  would attempt the illegal `requested → releasing` hop (`state.py` only allows
  `requested → released`) and call accounting. Instead add an explicit early branch in
  `_release_locked`: if `current.state is REQUESTED`, transition directly
  `requested → released`, audit, and return released — NO `releasing` hop, NO
  `stamp_active_ended`, NO `reconcile`, NO ledger credit (the row was never reserved).

- `request_allocation` threads `on_capacity` and the target descriptor into the domain request.
