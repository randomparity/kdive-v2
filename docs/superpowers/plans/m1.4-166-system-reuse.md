# Plan — M1.4 #166: system reuse in `runs.create`

ADR-0070 · Spec `docs/specs/m1.4-system-catalog-scheduling.md` ("Explicit system reuse").

## Problem framing

`runs.create(investigation_id, system_id, build_profile)` already attaches a Run to an
existing System and **never provisions** (provisioning is the separate `systems.provision`
step). So "skip provision only" is already structurally true. What #166 adds is the
*discover-and-validate* safety contract so an agent can attach to a `ready` System it did
**not** self-provision (found via `systems.list`, possibly from another Investigation).

## What is "the request" (Finding 1 — resolved from ground truth)

`runs.create` has **no size operand** — a Run carries a `build_profile`, not a size. So
"System sizing ≥ the request" is **not** a mandatory `runs.create` gate, and we do **not**
add required/optional sizing params that default to disabling a check.

Per **ADR-0070 §Decision lines 69-74**, the preconditions every `runs.create` enforces *in
general* (reuse or self-provisioned) are exactly the **three**:

1. Live allocation (`stale_handle`),
2. Single project (`configuration_error`),
3. One non-terminal Run per System (`transport_conflict`).

These are enforced **unconditionally** on every `runs.create`.

The **snapshot-≥ predicate** (ADR-0070 lines 58-67) is the *discovery/candidacy*
definition — "the shape *name* is a convenience filter, not the reuse identity." Primary
≥-discovery is **`systems.list`'s** job (already shipped, #159). We implement the predicate
as a single canonical, well-tested helper:

> `System snapshot sizing ≥ resolved tuple` **AND** `pcie_claim contains the required devices`

and **test the helper directly** (≥ matches; smaller fails; full-custom profile-only-sizing
case; pcie contains vs missing) — that satisfies the issue's "snapshot-≥ match honored"
acceptance.

### Optional reuse-match assertion on `runs.create` (Finding 2 — real consumer)

The helper is **not** orphaned: `runs.create` gains **optional** match-assertion params so
the agent can re-assert, under the lock, the requirements it discovered via `systems.list`
(closing the list→create TOCTOU):

- `require_vcpus: int | None`
- `require_memory_gb: int | None`
- `require_disk_gb: int | None`
- `require_pcie: list[str] | None` (vendor:device match specs)

These are **assertions, not a size operand**: there is no *required* sizing input, and the
three preconditions remain unconditional. When **any** is provided, `runs.create` reads the
System's persisted snapshot and calls the helper; a snapshot that does not satisfy
`≥`/`pcie-contains` (or a `class=` pcie spec, unsupported against claims) ⇒
`configuration_error`. When **all are omitted** (the self-provisioned attach), only the
three preconditions apply — backward compatible, every existing caller/test unchanged.

This makes "snapshot-≥ match honored" testable through the real tool **and** gives the
helper a production consumer (no unused-symbol guard risk). The `runs.create` schema
*does* change (four optional params) ⇒ regenerate generated docs + update
`tests/mcp/core/test_tool_docs.py`.

## Canonical snapshot sizing read (Finding 3)

One read with explicit MB↔GB normalization, reusing `_MB_PER_GB` and the `AllocationSizing`
shape from `systems/provision.py`:

- When the Allocation carries a complete snapshot (`requested_vcpus` /
  `requested_memory_gb` / `requested_disk_gb` all non-NULL) → use it (GB→MB via
  `_MB_PER_GB`).
- Else (full-custom / legacy: `requested_*` NULL) → read the System's
  `provisioning_profile` JSON (`vcpu` / `memory_mb` / `disk_gb`), which is always concrete
  (`_stored_profile_for` guarantees it at provision time).

Both normalize to an `AllocationSizing(vcpu, memory_mb, disk_gb)` so the ≥ compare is
unit-consistent. The match helper takes a resolved tuple + the System's `pcie_claim`; PCIe
match mirrors `systems.list`'s `_pcie_clause` (claims carry no `class_code`, so a `class=`
spec is `configuration_error`, only `vendor:device` is honored).

## Preconditions re-validated UNDER the lock (Finding 2 + 4)

Today `_create_locked` holds `SYSTEM` + `INVESTIGATION` and re-checks System state +
Investigation, but NOT the allocation under a lock, NOT one-Run-per-System.

**Lock order (corrected — CRITICAL).** The global total order is
`PROJECT < RESOURCE < ALLOCATION < SYSTEM` (then `INVESTIGATION → RUN`), per
`src/kdive/db/locks.py:25-29` and **ADR-0040 §1**. ALLOCATION must be acquired **before**
SYSTEM. The reconciler `→expired` sweep holds `PROJECT → ALLOCATION → SYSTEM`
(`reconciler/loop.py:215-216`) and `allocations.release` holds `PROJECT → ALLOCATION`
(`allocation_release.py:96-97`); a path that took SYSTEM then ALLOCATION would deadlock
against them.

So `runs.create` acquires **`ALLOCATION → SYSTEM → INVESTIGATION`** (allocation FIRST). The
ALLOCATION key must be known *before* the first lock: the current code only learns the
allocation id after locking SYSTEM (`create.py:72`). Rework: the pre-lock fetch already
reads `system` (`create.py:67`) and `alloc` (`create.py:72`) — carry `system.allocation_id`
from that pre-lock read into `_create_locked` as the ALLOCATION lock key, then under the
locks re-read the allocation by that id (authority) and re-read the System. Mirror
`reconciler/loop.py:215-218` and `allocation_release.py:96-97`. Documented in
`_create_locked`.

Under the lock, in order:

1. **System** — re-fetch; gone (`torn_down/failed/crashed`) ⇒ `stale_handle`; not `ready`
   ⇒ `configuration_error` (unchanged).
2. **Live allocation** — re-fetch under the ALLOCATION lock; `state != ACTIVE` ⇒
   `stale_handle`. **Also** (ADR lines 76-78): an `ACTIVE` allocation whose
   `lease_expires_at < now()` is the *expiring* orphan-reaping window ⇒ `stale_handle`.
3. **Single project** — System.project == Investigation.project ⇒ else
   `configuration_error` (the top-level pre-lock check stays as fast-fail).
4. **One Run per System** — count non-terminal Runs (`CREATED`/`RUNNING`) on the System;
   ≥1 ⇒ `transport_conflict`.
5. Investigation open/active (unchanged), then insert the Run.

The pre-lock checks at the top of `create_run` stay (fast-fail); the lock is the authority.
The race test drives two concurrent `runs.create` on one System → exactly one `created`, the
other `transport_conflict`.

## Skip-provision control flow

Nothing to skip — `runs.create` never enqueued a provision job. Acceptance "no provision job
runs" is asserted by `count(jobs WHERE kind='provision') == 0` after a reuse attach; "a
fresh System still provisions" is the existing `systems.provision` path, unchanged. The Run
still builds/installs/boots (existing `runs.build/install/boot`).

## Test matrix

Helper (direct):
- snapshot ≥ matches; strictly-bigger matches; smaller-on-any-axis fails.
- full-custom System (allocation `requested_*` NULL, size only in `provisioning_profile`)
  resolves + matches correctly.
- pcie: claim contains required `vendor:device` matches; missing device fails; `class=`
  spec rejected (`configuration_error`).

`runs.create` (integration):
- matching ready System → created, zero provision jobs, build still works.
- optional assertion satisfied → created; assertion miss (System smaller / missing pcie /
  `class=` spec) → `configuration_error`; omitted assertion → created (3 preconditions only).
- terminal alloc → `stale_handle`; ACTIVE-but-lapsed-lease alloc → `stale_handle`
  (re-checked under the ALLOCATION lock).
- cross-project → `configuration_error`.
- existing non-terminal Run on the System → `transport_conflict`.
- race (reuse vs reuse): two concurrent `runs.create` same System → one created, one
  `transport_conflict` (proves the per-System / per-Allocation lock serializes).
- race (reuse vs reconciler/release): `runs.create` interleaved with the
  `PROJECT→ALLOCATION→SYSTEM` expired sweep (or `allocations.release`) does not deadlock and
  ends in a consistent state — the cross-path cycle the `ALLOCATION→SYSTEM` order must
  survive (reuse-vs-reuse alone won't exercise it).
- existing tests (first-run flip, gone-system, non-active-alloc) stay green.
