# Plan â€” #161: shape in allocation selector; size flows into provisioning

## Goal
Unify the allocation selector and the provisioning-profile sizing through the shape
resolver (ADR-0067 + ADR-0024 delta). A request names `shape: NAME` **XOR** full-custom
`{vcpus, memory_gb, disk_gb}`; the resolved size is the single authority for pricing,
the per-host capacity check, and the booted libvirt domain. Persist the sizing-snapshot
identity (`allocations.shape`, `systems.shape`, `allocations.requested_disk_gb`).

### 7. Provisioning profile sizing delta (`profiles/provisioning.py` + `systems/provision.py`)

**Schema change.** `ProvisioningProfile.vcpu/memory_mb/disk_gb` â†’ `int | None = None`; a
present value must still be `> 0` (field validator), preserving the old
reject-non-positive behavior. A profile may now omit sizing â€” but a *stored* profile must
never have NULL sizing, because `render_domain_xml` dereferences `profile.memory_mb` /
`profile.vcpu` unconditionally (Finding 2). Reconciliation guarantees concreteness before
the row is written.

**Reconciliation helper.** New `reconcile_profile_sizing(profile_input, *, vcpus,
memory_mb, disk_gb)` in `profiles/provisioning.py`:
- takes the raw profile **dict** (`ProvisioningProfileInput`) and the Allocation snapshot;
- for each of `vcpu` / `memory_mb` / `disk_gb`: if absent, fill from the snapshot; if
  present and â‰  the snapshot value, raise `configuration_error` (conflicting restatement);
- returns a new dict (**build, not mutate**) with all three concrete, ready to `parse`.
- It reads only the passed snapshot ints â€” never the catalog/`resolve_shape` â€” so a
  `shapes.set` after stamping cannot re-size (Finding 2, snapshot-not-catalog).

**Exact sequence â€” resolves the parse-before-lock ordering (Finding 1).** Today
`_create_for_allocation` parses the submitted profile at ~line 226, *before*
`_locked_allocation_system` loads the `alloc` (~line 319). The snapshot is unavailable
there. New two-stage flow, one single reconciliation point per create:
1. `_create_for_allocation`: **structural pre-parse** of the submitted profile (sizing now
   optional) for early provider/rootfs validation. Do NOT yet require sizing.
2. Inside the lock, once `alloc` is in scope: if the snapshot is complete
   (`requested_vcpus`/`requested_memory_gb`/`requested_disk_gb` all present),
   `reconcile_profile_sizing(submitted_dict, ...snapshot)` â†’ re-`parse` the reconciled
   dict â†’ that concrete profile is what `dump_profile` stores. Legacy allocations (NULL
   disk snapshot) skip reconciliation and keep profile-authoritative sizing (the submitted
   profile must then carry its own sizing or its loose-parse-then-store yields NULL â€” so
   for the legacy lane sizing stays *required at store time*: if reconciliation is skipped
   AND the parsed profile has any NULL sizing â†’ `configuration_error`).

This makes the **single reconciliation point** the create-insert path (both `provision`
and `define`), before `dump_profile`. The stored profile is therefore always concrete.

**`provision_defined` is a pure re-validate (Finding 2).** `_provision_defined_response`
re-parses the *already-concrete stored* profile (~line 459) and does **not** reconcile â€”
the sizing was frozen at `define`. A `shapes.set` between `define` and `provision_defined`
has no effect: provisioning reads the stored profile, never the catalog.

- `render_domain_xml` reads `profile.memory_mb`/`profile.vcpu`; the stored profile always
  carries concrete sizing, so boot is unaffected.
