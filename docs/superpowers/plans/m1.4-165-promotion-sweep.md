# Plan — #165 reconciler work-conserving FIFO promotion sweep + queue_timeout reaper

ADR-0069 · Spec m1.4-system-catalog-scheduling.md · builds on #164 (queue) + #162 (PCIe).

## Goal

Add a reconciler pass that promotes the oldest *placeable* `requested` allocation per
resource to `granted` (work-conserving), re-running PCIe-aware, cordon-skipping selection
from persisted inputs, replaying the existing check-then-debit under `PROJECT → RESOURCE`.
A budget-recheck failure at promotion terminates the request (`failed`); a request never
placeable past a max-wait window is reaped to `failed(queue_timeout)` — a new distinct
`ErrorCategory`.

## Key facts established by reading the code

- `requested` rows persist: size (`requested_vcpus/memory_gb/disk_gb`), `shape`,
  `requested_pcie_specs`, and exactly one of `requested_kind` / `requested_resource_id`.
  **No `window` column** — promotion re-resolves the default window via
  `resolve_window_hours(None)` (window was never persisted in #164; default is correct).
- Occupancy predicate `_OCCUPYING = GRANTED/ACTIVE/RELEASING` excludes `requested`, so a
  candidate never self-blocks its own promotion replay.
- The synchronous grant inserts a NEW row (`_grant`); promotion **transitions an existing
  row** (stamp `resource_id` + lease, reserve, `requested → granted`). So `_grant` (insert)
  is not reusable verbatim — but the **gate** is: `_within_alloc_quota`, `within_budget`,
  `_host_cap_check`, `_resolve_pcie_claim`, and the pricing (`resolve_coeff/rate/cost/
  quantize_kcu`) are reused. I extract a shared `price_estimate` + `replay_capacity_gate`
  helper from admission so both paths run identical checks (no forked gate). The gate
  returns a **structured outcome carrying the existing `queueable` flag** (and the claimed
  PCIe devices on success), NOT just a category: the budget denial and the host-cap denial
  both have `category=ALLOCATION_DENIED` (see admission lines 245-251 vs 405), so terminate-
  vs-wait MUST branch on `queueable`/reason, never on category. **Only a budget recheck
  failure terminates** (`queueable=False`, `category=ALLOCATION_DENIED` from the budget
  branch) → `failed` — this matches ADR-0069 "a budget recheck failure at promotion
  TERMINATES." Every other denial (host-cap full, quota full, PCIe-busy CAPACITY, and PCIe-
  config CONFIG i.e. no schedulable host currently has the card) is a **wait** → stay
  `requested`; the queue_timeout reaper terminates the permanently-unplaceable ones after
  max-wait. Treating PCIe-config as a wait (not an immediate terminate) is deliberate: a
  cordon could lift or a host be added, and the ADR reserves immediate termination for the
  budget case only. The gate distinguishes the budget terminate from the rest because the
  budget denial is the unique `ALLOCATION_DENIED` with `queueable=False` AND no `reason`
  (host-cap carries `reason="at_capacity"`).
- Lock order: promotion takes **`PROJECT → RESOURCE → ALLOCATION`** (the global order
  `PROJECT → RESOURCE → ALLOCATION → SYSTEM`). Verified the only acquirers of two of
  {PROJECT, RESOURCE, ALLOCATION} are: admit `PROJECT → RESOURCE` (same prefix, no
  inversion); release/expiry/renew `PROJECT → ALLOCATION` or `PROJECT` only — **none take
  RESOURCE** (`rg LockScope.RESOURCE` hits only admission + resource_discovery, the latter
  takes RESOURCE alone). resource_discovery takes RESOURCE with no PROJECT/ALLOCATION, so
  it can never form a cycle with promotion. Therefore `PROJECT → RESOURCE → ALLOCATION` has
  no deadlock against any path: every two-lock path acquires PROJECT first and ALLOCATION
  last, and promotion's intervening RESOURCE is never contended by a path that already holds
  ALLOCATION. ALLOCATION fences the released-while-queued race (same as the expiry sweep);
  RESOURCE is the capacity lock admit also uses, so promotion and a concurrent synchronous
  admit on the same host serialize on RESOURCE — no double-grant of the freed slot.
- Audit attribution: `record_system` hardcodes `agent_session=None`. Extend it with an
  optional `agent_session` param (default None preserves reconciler-teardown behavior) so
  the promotion `->granted` row carries the queued row's original `(principal,
  agent_session)`.
- `allocations` has no `failure_category` column; `queue_timeout` is carried in the audit
  args + log, transition `requested->failed`.

## Selection algorithm (work-conserving oldest-placeable)

Per pass:
1. Candidate-select all `requested` rows ordered by `created_at` (the partial index), oldest
   first, with their persisted inputs.
2. Walk oldest-first. For each, **in its own committed transaction** (like `_expire_one`),
   re-resolve the schedulable candidate hosts from persisted inputs (by-id → that host if
   schedulable; by-kind → all `available AND NOT cordoned` hosts of the kind, oldest-first),
   PCIe-filtered. Attempt to promote onto the first host that passes the full gate under
   `PROJECT → RESOURCE → ALLOCATION`. On success → granted, commit, count it, continue.
   Per-candidate commit is load-bearing: each candidate re-reads the **post-commit**
   occupancy/quota counts of its predecessors, so the per-project grant quota and per-host
   cap cannot be overshot within one pass (a single shared transaction would let N
   candidates all observe the same pre-promotion count).
3. Because we scan oldest-first and each request independently re-resolves *all* matching
   hosts, a younger request placeable on a free host B is promoted even while the global-
   oldest waits on busy host A: the oldest simply fails to place (its host A is full) and we
   move on; the younger one places on B. This is exactly "oldest *placeable* per resource."
4. Budget recheck failure at promotion → `requested → failed` (terminates, no re-queue).
   PCIe-busy / host-cap-full / quota-full / PCIe-config → leave `requested` (waits).
   Per-allocation isolation: a candidate whose transaction raises rolls back and stays
   `requested` for the next pass; it never starves siblings (mirrors `_sweep_expired`).
5. Reaper runs **after** the promotion step in the same pass, so every aged row already had
   its placement chance this pass. It candidate-selects `requested AND created_at < now() -
   max_wait`, then per row, under `PROJECT → ALLOCATION`, **re-reads and re-validates BOTH
   the `requested` state AND the `created_at < now() - max_wait` age** before flipping
   `requested → failed` with `queue_timeout`. The locked age re-check means a row that was
   promoted by step 2 (now `granted`) is skipped, and a row that is currently placeable but
   merely lost the scan this pass is NOT reaped on age alone — it is reaped only once it has
   genuinely never placed past the window (the next pass gives it another chance first).

Max-wait window: module constant `DEFAULT_QUEUE_MAX_WAIT = timedelta(hours=24)` mirroring
the other reconciler `DEFAULT_*` knobs, threaded through `reconcile_once` / `Reconciler`.

- freed slot promotes oldest placeable + reserves at grant + sets lease + stamps resource_id
- work-conserving: global-oldest waits on busy host A, younger placeable on free host B is
  promoted first
- over-budget-at-promotion → failed (not re-queued, not retried into a loop)
- a request that is BOTH host-cap-full AND over budget is terminated (budget) not parked —
  proves terminate-vs-wait branches on `queueable`, not on the shared `ALLOCATION_DENIED`
- per-project grant quota headroom = 1 with two placeable queued rows → exactly one promoted
  (proves per-candidate committed transaction; the second stays `requested`)
- never-placeable past max-wait → failed(queue_timeout) — assert category, NOT lease_expired
- a request past max-wait that IS placeable this pass is promoted, not reaped (reaper runs
  after the promotion step; the aged-but-placeable row grants)
- cordoned host skipped (stays requested)
- PCIe-aware promotion resolves+claims a device at grant (reuse #162); PCIe-busy request
  stays requested (not failed) until a device frees
- grant audit attributed to original (principal, agent_session)
- lock sharing: a release that pre-holds PROJECT→ALLOCATION on the queued row blocks the
  sweep; the row goes released; the sweep then skips it (no promote of a released row)
- concurrent synchronous admit vs sweep can't double-grant the freed slot (RESOURCE lock)
