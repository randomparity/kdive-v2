# M1.1 — Platform-scoped RBAC Tier Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: use superpowers:subagent-driven-development
> (recommended) or superpowers:executing-plans to implement each sub-issue. Each
> sub-issue below is sized for a single PR; its bite-sized TDD steps are authored at
> execution time. Steps within an issue use checkbox (`- [ ]`) tracking.

**Goal:** Add the platform-scoped authorization tier from
[`../specs/m1.1-platform-rbac-tier.md`](../specs/m1.1-platform-rbac-tier.md) — a
`platform_roles` claim orthogonal to the per-project `roles` map, the
`require_platform_role` seam, a `platform_audit_log` read-access trail, and the one
cross-project read with a live consumer, `accounting.report`.

**Architecture:** Unchanged core. M1.1 adds a *second, independent* authorization axis
alongside the per-project one: `require_role` (per-project, ranked) is untouched;
`require_platform_role` (platform, independent grants with a single `admin ⊇ auditor`
partial order) is added beside it. The new read-access audit cannot reuse the per-project
`audit_log` (its `project`/`object_*` are `NOT NULL` and `audit.record` enforces
`project in ctx.projects`), so a dedicated `platform_audit_log` + `audit.record_platform`
writer lands. `accounting.report` composes existing ledger reads across projects; its
granted-set form rides the existing `require_role`, not the new seam.

**Tech Stack:** As M1 — Python 3.13 · `uv` · FastMCP · `psycopg` 3 (async) · Pydantic v2 ·
`ruff`/`ty`/`pytest` · `prek`. No new runtime dependency. No VM (disposable Postgres only).

## Phase A — The seam

### Issue P1 — Platform RBAC seam (#96)
- **Labels:** `area:security`
- **Depends on:** — (M1 complete)
- **Goal:** The platform authorization axis and its read-access trail; no tool yet beyond a
  test harness.
- **Files:** Extend `src/kdive/security/authz/rbac.py`, `src/kdive/security/audit.py`,
  `src/kdive/mcp/auth.py`; create `src/kdive/db/schema/0005_platform_audit_log.sql`; tests
  under `tests/security/`, `tests/mcp/`, `tests/db/`.
- **Scope:**
  - `rbac.py`: `class PlatformRole(StrEnum)` = `PLATFORM_ADMIN`/`PLATFORM_OPERATOR`/
    `PLATFORM_AUDITOR`. `platform_roles_from_claims(claims)` reads the **flat array** claim
    `platform_roles` → `frozenset[PlatformRole]`; absent → empty set; a non-array or an
    unknown value raises `AuthError` (fail closed, mirroring `roles_from_claims`).
    `require_platform_role(ctx, role)` encodes the partial order — each role satisfies
    itself, `PLATFORM_ADMIN` satisfies `PLATFORM_AUDITOR`, `PLATFORM_OPERATOR` satisfies
    neither — and raises `AuthorizationError` (same type/handler mapping as `require_role`)
    on failure. This is the sole enforcement seam **for platform roles**; `require_role`
    stays unchanged and unaware of platform roles.
  - `mcp/auth.py`: add `platform_roles: frozenset[PlatformRole] = frozenset()` to
    `RequestContext` (`compare=False`, like `roles`); populate it in the context builder via
    `platform_roles_from_claims(claims)` next to the existing `roles=roles_from_claims(claims)`.
  - `0005_platform_audit_log.sql`: append-only
    `platform_audit_log(id uuid PK default gen_random_uuid(), ts timestamptz NOT NULL default now(),
    principal text NOT NULL, agent_session text, platform_role text, tool text NOT NULL,
    scope text NOT NULL, args_digest text NOT NULL)`. **`platform_role` is NULLABLE** (null for
    an audited granted-set member read that carries no platform role; ADR-0043 §4). Forward-only,
    additive (ADR-0015); a no-op on re-run.
  - `audit.py`: `record_platform(conn, *, principal, agent_session, platform_role, tool, scope, args)`
    INSERTs one `platform_audit_log` row (reusing `args_digest`) on `conn` without opening a
    transaction (caller composes it). **No `project in ctx.projects` guard** — platform
    authority is project-independent, so a principal with empty `ctx.projects` writes a row.
    Used for successful platform reads, audited granted-set member reads, and
    `require_platform_role` denials.
- **Acceptance** (copied to #96):
  - Claim parsing: a valid array parses; absent → empty set; a non-array claim and an unknown
    role each raise `AuthError`.
  - `require_platform_role`: each role satisfies itself; `platform_admin` satisfies
    `platform_auditor`; `platform_operator` satisfies neither; an empty `platform_roles` is
    denied.
  - The `0005` migration creates `platform_audit_log` and is idempotent on re-run.
  - `record_platform` writes a row for a principal with **empty `ctx.projects`** and for a
    denial (proving the membership guard is bypassed).
  - `RequestContext` built from a token with a `platform_roles` claim carries the parsed set;
    a token without it carries the empty set.
  - No tool calls `record_platform` / `require_platform_role` yet beyond a test harness.

## Phase B — The first cross-project read

### Issue P2 — `accounting.report` (#97)
- **Labels:** `area:allocation` · `area:mcp-api`
- **Depends on:** P1 (#96)
- **Goal:** A multi-project usage/billing rollup + per-allocation reserved/reconciled
  variance, optional `group_by=principal` / time-window, in two scope forms.
- **Files:** Extend `src/kdive/domain/accounting.py`, `src/kdive/mcp/tools/accounting.py`;
  tests under `tests/domain/`, `tests/mcp/`.
- **Scope:**
  - `accounting.py` (domain): a `report(conn, *, projects, group_by, window)` rollup that, for
    the given project set, returns per-project `{reserved, reconciled, variance}` (variance =
    `reconciled − reserved` from the signed ledger) plus a total. With `group_by="principal"`,
    group by `allocations.principal` via `ledger ⋈ allocations` on `allocation_id` (the ledger
    has no `principal` column; the existing `ledger_allocation_id_idx` covers the join). With a
    `window`, bound by `ledger.ts`. Reuses the existing `_by_cost_class`/`_sum_scalar` helpers'
    SQL shape; no schema change beyond P1's table.
  - `mcp/tools/accounting.py`: a `report(pool, ctx, *, scope, projects, group_by, window)` tool
    with two scope forms:
    - **`scope="granted-set"`** — resolve the target set, distinguishing default from named
      (membership ≠ role in this codebase: a project can be in `ctx.projects` with **no** role,
      and `require_role` *raises* on a role-less membership — `security/authz/rbac.py:85-91`,
      `roles_from_claims` "membership without a role"):
      - **default (no `projects` arg)** — the set is the projects in `ctx.projects` where
        `ctx.roles.get(p)` ranks **≥ `viewer`**; role-less memberships are **filtered out**, not
        raised on (so a manager who is `viewer` on A/B but a bare member of C gets A/B, not an
        error). A set resolving to zero projects → an **empty rollup** (success, not error).
      - **named (`projects` arg given)** — authorize each with `require_role(ctx, p, Role.VIEWER)`,
        which **raises** `authorization_error` for a non-member **or** role-less project (the
        caller named it explicitly, so a typo/over-reach surfaces rather than being silently
        dropped).
      **Audit by read-shape:** call `audit.record_platform` (with `platform_role=None`) **iff**
      the resolved set spans >1 project **or** `group_by="principal"`; a single-project ungrouped
      read writes no audit row.
    - **`scope="all-projects"`** — `require_platform_role(ctx, PlatformRole.PLATFORM_AUDITOR)`
      (satisfied by `platform_admin`); rollup over every project (the project universe =
      `SELECT DISTINCT project FROM budgets`); **always** `audit.record_platform` (with the
      granting `platform_role`) on success. **A denial by a principal that holds ≥1
      `platform_role` is audited** (ADR-0043 §4): catch the `AuthorizationError`, open a
      connection, `audit.record_platform(..., platform_role=<the held platform role(s)>,
      scope="all-projects")` the denial, then map to `authorization_error`. (The role check, like
      `require_role` in `usage`/`estimate`, runs before any pool connection is open, so the
      denial-audit must open its own connection in the `except` path.) **A denial by a principal
      with no `platform_roles` is *not* audited** — that is the routine "wrong tool, no platform
      grant" case, and auditing it would let any authenticated token amplify writes into
      `platform_audit_log` (the tool is an openly-registered read); the accountability target is a
      *platform principal* over-reaching its grant (e.g. `platform_operator` attempting the
      auditor read), which the held-role check captures.
    - Both forms return per-allocation variance + the rollup; `scope` recorded in the audit
      `scope` column (`all-projects` or the project set). Register as `accounting.report` in
      `register(...)` beside the existing tools.
- **Acceptance** (copied to #97):
  - **all-projects:** a `platform_auditor` token gets a rollup spanning ≥2 projects with
    per-project + total variance matching a hand-computed ledger; the call writes a
    `platform_audit_log` row (with `platform_role` set), not an `audit_log` row. A denial by a
    principal **holding ≥1 `platform_role`** (e.g. a `platform_operator` attempting the auditor
    read) writes a `platform_audit_log` denial row recording the held role(s); a denial by a
    principal with **no `platform_roles`** (a project-only `viewer`/`operator`/`admin` token)
    writes **no** row (routine non-grant, not an accountability event — avoids unprivileged
    write-amplification on this openly-callable read).
  - **granted-set (default):** a token that is `viewer` on A+B and a **bare member** of C (no
    role on C, no platform role) gets a rollup over exactly **A+B** — the role-less C is filtered
    out, not raised on; a token whose memberships all lack a role resolves to an empty rollup
    (success).
  - **granted-set (named):** a request naming a non-member **or** role-less project →
    `authorization_error` (not dropped); a zero-project resolution → empty rollup (success); an
    audit row is written **iff** >1 project or `group_by=principal` (with `platform_role` null),
    and **not** for a single-project ungrouped read; never a per-project `audit_log` row.
  - **group_by=principal:** over a window, per-principal totals via `ledger ⋈ allocations`
    match a hand-computed expectation, in both scope forms.
  - **separation-of-duties negatives:** a project-only `viewer`/`operator`/`admin` token is
    denied the **all-projects** form; a `platform_operator` token is denied the auditor read.
  - No VM needed (disposable Postgres).
