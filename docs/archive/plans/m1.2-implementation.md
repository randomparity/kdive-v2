# M1.2 — Live-stack End-to-End Validation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: use superpowers:subagent-driven-development
> (recommended) or superpowers:executing-plans to implement each sub-issue. Each
> sub-issue below is sized for a single PR; its bite-sized TDD steps are authored at
> execution time. Steps within an issue use checkbox (`- [ ]`) tracking.

**Goal:** Stand up the first executable, real-infrastructure end-to-end test from
[`../specs/m1.2-live-stack-e2e.md`](../specs/m1.2-live-stack-e2e.md) — drive the full spine
(allocate → provision → build → install → boot → attach → force-crash → capture vmcore →
release → report) **over the live MCP HTTP stack** under real OIDC tokens, on a host with
real libvirt, replacing the unimplemented M0 walking-skeleton stub.

**Architecture:** No product code changes — the driver exercises shipped handlers unchanged.
M1.2 adds a *test-side* wire client (`fastmcp.Client` over streamable HTTP), a real
OIDC-token helper (mock-oauth2-server → real JWKS/`JWTVerifier`), a phase-structured spine
driver, and stack orchestration. The kdive `server`/`worker`/`reconciler` run **on the host**
against the existing containerized backends (Postgres + MinIO + mock-OIDC); containerizing
them is a deferred follow-on (sub-issue F). The one tool the driver needs beyond M0/M1,
`accounting.report`, comes from **M1.1 P2** ([#97](https://github.com/randomparity/kdive/issues/97)).

**Tech Stack:** As M1 plus `fastmcp.Client` (already a dependency, client side) and the
existing `docker-compose.yml`. No new runtime dependency. The driver + report assertions need
a KVM host (`live_stack` marker). The harness's in-memory smoke tier + orchestration are
CI-able on `pull_request`; the harness's real-server smoke tier (and the spine/report) need
the stack stood up.

## Test environments

- **A + C** carry **no VM**. A's **in-memory** smoke tier runs on the M1 service harness
  (disposable Postgres + MinIO + mock-OIDC) on `pull_request`; A's **real-server** smoke tier
  connects to a host-run `server` + real issuer over HTTP and runs under `live_stack` once the
  stack is up.
- **D + E** require the M0/M1 **KVM/nested-virt host** with the kdump-enabled guest image and
  pinned kernel tree (`KDIVE_GUEST_IMAGE`, `KDIVE_KERNEL_SRC`), **plus a reachable stack**
  (`KDIVE_STACK_BASE_URL`, the OIDC issuer). The `live_stack` preflight `pytest.skip`s with an
  actionable reason when fixtures **or** the stack are absent (the ADR-0035 §4 idiom).
- **A's gate:** confirm the mock-oauth2-server can mint both the nested-object `roles` claim
  and the `platform_roles` array claim through the real verifier **before D is scheduled** (the
  open assumption in ADR-0042 §3); if it cannot, A redesigns token acquisition first.

## Phase A — Wire harness + real tokens

### Issue A — MCP-over-HTTP test harness + OIDC token issuance (#98)
- **Labels:** `type:test` · `area:mcp-api` · `area:security`
- **Depends on:** — (no VM, no other sub-issue)
- **Goal:** A reusable `fastmcp.Client` wrapper and an OIDC token helper, validated through the
  server's real JWKS/`JWTVerifier`, in **two tiers**: an **in-memory-client smoke test that is
  CI-able** on `pull_request` (no running server process), and a **real-server/issuer smoke
  test** that needs the stack stood up (sub-issue C) and is operator-run, gated `live_stack`.
  The real-issuer claim-shape gate (ADR-0042 §3) lives in the second tier.
- **Files:** Create `tests/integration/live_stack/harness.py`; register the `live_stack` marker
  in `pyproject.toml`; smoke test under `tests/integration/`.
- **Scope:**
  - `harness.py`: a `LiveStackClient` wrapping `fastmcp.Client` (streamable HTTP) against
    `KDIVE_STACK_BASE_URL`, exposing typed `call_tool(name, **args)` returning the parsed
    envelope; and a `mint_token(role=..., projects=..., platform_roles=...)` helper that
    obtains a bearer token from the mock-oauth2-server carrying the **nested-object `roles`
    claim** (`{<project>: <role>}`) and, when requested, the **`platform_roles` array claim** —
    not the in-process `mint`. The server validates them through its configured `JWTVerifier`
    against the issuer JWKS.
  - `pyproject.toml`: add the `live_stack` marker (distinct from `live_vm`: needs a running
    stack + issuer, not just a KVM host).
  - Smoke test, two tiers (both: connect → `list_tools` → one read-only call per role
    `viewer`/`operator`/`admin` + one `platform_auditor` read-only call; **no VM**):
    - **CI-able tier** — in-memory FastMCP client + injected verifier (the ADR-0035 §3 token
      path), runnable on `pull_request`; exercises registration/schemas/envelopes, **not** the
      real JWKS or transport.
    - **`live_stack` tier** — `fastmcp.Client` over HTTP against a host-run `server` + Postgres
      + real issuer (needs sub-issue C's stack); this is the tier that exercises the real
      JWKS/`JWTVerifier` path and the claim-shape gate. Skipped on `pull_request`.
- **Acceptance** (copied to #98):
  - CI-able tier: the in-memory smoke test passes on `pull_request` (no server process).
  - `live_stack` tier: the issuer mints the nested-object `roles` claim and the `platform_roles`
    array claim; all tokens validate through the server's **real** verifier over HTTP (the gate
    confirming ADR-0042 §3's open assumption); passes against a locally-run stack.
  - `harness.py` is importable by D.
  - If the issuer cannot mint that claim shape, A redesigns token acquisition before D is
    scheduled (documented in the PR).

## Phase C — The spine

### Issue D — Phase-structured spine driver (replaces the stub) (#100)
- **Labels:** `type:test` · `provider:local-libvirt` · `area:mcp-api`
- **Depends on:** A (#98), C (#99)
- **Goal:** Drive the full spine over the wire and assert the protocol/#1/#2/#5/RBAC/#3 exit
  criteria; a failure names its phase.
- **Files:** Create `tests/integration/test_live_stack.py`; **delete**
  `test_walking_skeleton_full_path` from `tests/integration/test_walking_skeleton.py`.
- **Scope:**
  - `test_live_stack.py` (marker `live_stack`, preflight skip): advances through named phases
    via A's harness — `allocate → provision → open-investigation → create-run → build →
    install → boot → attach → crash → capture → introspect → release → report` — each step a
    tool call over HTTP under a specific role; the async job kinds (`provision`/`build`/
    `install`/`boot`/`capture_vmcore`) are enqueued by the caller's call and drained by the
    **real** host `worker` + `reconciler`, the driver polling `runs.get`/`systems.get` to
    `succeeded` before each dependent phase. Each phase records pass/fail and threads its output
    id forward.
  - **Setup prerequisite (`crash` phase):** the `provision` profile must opt into `force_crash`
    (`provider.local_libvirt.destructive_ops`) and the run must carry the destructive
    capability scope, so the 3-check gate (admin role ∧ capability scope ∧ profile opt-in)
    passes at `crash`. Established up front, not discovered mid-spine.
  - Per the replace-don't-deprecate rule, the `live_vm` full-path stub
    (`test_walking_skeleton_full_path`) is deleted, not left as a placeholder. The non-gated
    walking-skeleton tier and the M1 `live_vm` smoke tests keep the `live_vm` marker untouched.
- **Acceptance** (copied to #100):
  - On a KVM host with fixtures + stack, the spine reaches `release`; a failure names its phase.
  - **Protocol:** every step returns a well-formed MCP envelope over HTTP; tokens validate
    through the real JWKS path.
  - **#1** a fetchable redacted vmcore artifact exists in MinIO at the end.
  - **#2** an `audit_log` row for every transition + the `force_crash`, under the request's
    `(principal, agent_session, project)` tuple.
  - **#5** after release the System is `torn_down` **and** `Discovery.list_owned()` returns no
    `OwnedInfra` for the released `system_id`.
  - **RBAC negatives** over the wire: a `viewer` token is denied operator/admin operations;
    `force_crash` is denied without `admin`.
  - **#3 redaction:** a planted secret does not leak through the wire path (transcript +
    artifact-sensitivity).

## Phase D — Accounting assertions

### Issue E — Accounting assertions + report artifact (#101)
- **Labels:** `type:test` · `area:allocation`
- **Depends on:** M1.1 P2 (#97), D (#100)
- **Goal:** The `report` phase — drive `accounting.report` over the wire, assert the run's
  spend, emit the report artifact. Separate from D for bisectability.
- **Files:** Extend `tests/integration/test_live_stack.py` (the `report` phase + assertions).
- **Scope:**
  - Drive `accounting.report`'s **all-projects** form with a `platform_auditor` token over the
    wire (exercising the M1.1 platform claim end-to-end), alongside the existing `viewer`
    `accounting.usage(investigation_id=…)`. Assert ledger `reserved`/`reconciled`/variance for
    the run and emit the report artifact.
  - The spine runs a **single project**, so E does **not** validate the multi-project rollup or
    the granted-set form — that correctness is M1.1 P2's in-process test (≥2 projects); E proves
    wire reachability and the platform-vs-project authorization boundary.
- **Acceptance** (copied to #101):
  - The report reflects the run's real spend and is written as an artifact.
  - `accounting.report` (all-projects form) is reachable over the wire under a
    `platform_auditor` token and denied to a project-only token.
